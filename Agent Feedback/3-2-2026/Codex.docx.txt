Findings
1. Lack of formal schema definition: The specification describes categories textually but provides no machine-readable JSON Schema, potentially causing implementation variances and parsing errors. Severity: High (impacts interoperability and reliability).
2. Undefined conflict resolution for multi-authority signatures: Multi-authority verification is outlined, but no procedures for resolving contradictory signatures from different authorities. Severity: High (risks trust erosion and decision paralysis).
3. Incomplete revocation mechanisms: Revocation status is mentioned in verification section, but no detailed protocol for signature revocation or authority disqualification. Severity: Medium (could allow persisted invalid metadata).
4. Insufficient handling of stale metadata: TTL is defined, but no required behaviors for consumers when metadata expires or update mechanisms fail. Severity: Medium (may lead to use of outdated information).
5. Limited guidelines for extensibility: Extensibility allows custom facts, but lacks requirements for documenting or versioning extensions, risking ecosystem fragmentation. Severity: Low (affects long-term maintainability but not immediate functionality).

Edit 2: Define Conflict Resolution Procedure
* Change text: Add new subsection 8.7 after 8.6: 8.7 Conflict Resolution in Multi-Authority Verification When signatures from multiple authorities conflict (e.g., differing confidence levels or claims), consumers must apply the following deterministic procedure:
1. Aggregate signatures by section.
2. Select the signature with the highest confidence level from whitelisted authorities.
3. If ties occur, prioritize the most recent timestamp.
4. If unresolved, reject the section and fall back to unverified status. Implementations must log conflicts and notify metadata owners.
* Why it helps: Establishes a clear, deterministic process to handle discrepancies, maintaining trust and operational consistency.
* Compatibility impact: None (additive guidance; does not alter existing verification mechanics).
* Abuse risk impact: Mitigates abuse by preventing exploitation of conflicting signatures to inject false data.
Edit 3: Specify Revocation Protocol
* Change text: Expand section 5.2.9 (Verification) with: Revocation must follow this protocol:
o Authorities issue revocation notices as signed JSON objects containing the original signature ID, revocation timestamp, and reason code (e.g., "compromised", "expired").
o Consumers query a designated revocation endpoint (e.g., authority's /revocations endpoint) during validation.
o Metadata includes a "revocation_check_url" field pointing to the authority's revocation list. Revoked signatures invalidate the associated section.
* Why it helps: Provides a standardized method to invalidate compromised signatures, enhancing security and trust over time.
* Compatibility impact: Low (additive fields and checks; existing metadata without revocation URLs assumes no revocations).
* Abuse risk impact: Lowers risk by enabling quick invalidation of abused or fraudulent signatures.
Edit 4: Add Stale Metadata Handling Requirements
* Change text: Add to section 8.3 (Update Mechanisms): Consumers must handle stale metadata as follows:
o If TTL expires and no update is available, mark the metadata as "stale" and restrict usage to read-only or low-trust operations.
o Implementations must attempt at least one pull update before declaring staleness.
o Owners must provide an "update_endpoint" URL in core_identity for automated refreshes.
* Why it helps: Ensures consistent behavior when metadata ages, preventing reliance on outdated information.
* Compatibility impact: Low (additive field and behaviors; existing systems can ignore without breaking).
* Abuse risk impact: Reduces risk of abusing expired metadata for persistent false claims.
Edit 5: Provide Extensibility Guidelines
* Change text: Expand section 5.2.10 (Extensibility) with: Extensions must adhere to:
o Prefix custom keys with "x-" (e.g., "x-custom-fact").
o Include an "extension_version" field for each custom section.
o Document extensions in a linked "extension_docs_url" field.
o Maintain backward compatibility by not removing existing keys in updates.
* Why it helps: Standardizes extension practices, reducing fragmentation and improving long-term ecosystem coherence.
* Compatibility impact: None (additive rules; existing extensions remain valid).
* Abuse risk impact: Minimizes abuse by requiring documentation, deterring hidden malicious extensions.
