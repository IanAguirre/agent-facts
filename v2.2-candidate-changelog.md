High-Confidence Additions (v2.2)

1) Core Required Fields (Normative)
Change text (ready-to-paste):
```
Core Required Fields (Normative)
Every Agent-Facts page/document MUST include these top-level fields (visible and machine-readable):
- `protocol_version` (string, semantic version)
- `entity_legal_name` (string)
- `primary_url` (string, absolute URL)
- `last_modified_iso8601` (string, RFC3339 / ISO 8601 UTC, e.g. 2026-03-02T14:30:00Z)
- `jurisdiction` (string)
- `primary_contact` (object: { `phone`?, `email`? })

Missing any required field MUST cause validation to fail with a clear diagnostic code.
```
Why it helps: deterministic parsing and field completeness.
Compatibility impact: low (additive); non-compliant pages flagged but still readable.
Abuse risk impact: medium reduction (reduces ambiguous claims).
Acceptance test: validator fails when any required field is missing.

2) Timestamp Precision & Parity (Normative)
Change text:
```
Timestamps: `last_modified_iso8601` MUST use RFC3339/ISO8601 with UTC (Z suffix) to second precision. The HTTP `Last-Modified` header (if present) MUST match the HTML/JSON timestamp to second precision. Parsers SHOULD treat the payload stale if timestamps diverge.
```
Why it helps: removes timezone ambiguity; enables exact freshness checks.
Compatibility impact: low.
Abuse risk impact: reduces backdating and stale-data attacks.
Acceptance test: parser rejects timestamp without `Z` or mismatched HTTP header.

3) Claim Provenance Labels (Normative)
Change text:
```
Claim Provenance (Normative)
Each machine-readable claim MUST include provenance fields: `claim_id`, `claim_text`, `source_type` (one of: `official_record`, `first_party`, `third_party`, `self_asserted`), and optional `source_url` and `evidence_date`. Claims with `self_asserted` MUST be labeled as such in visible text.
```
Why it helps: distinguishes verified facts from self-assertion, reducing hallucination risk.
Compatibility impact: low (additive).
Abuse risk impact: high reduction for unverifiable claims.
Acceptance test: any `self_asserted` claim is emitted with explicit label.

4) Cross-file Consistency (Normative)
Change text:
```
Cross-File Consistency
`protocol_version` and `last_modified_iso8601` MUST match across the site assets: agent-facts HTML/JSON, `llms.txt`, and spec metadata endpoints. If mismatch occurs, consumers MUST treat the newest RFC3339 timestamp as authoritative and log the discrepancy.
```
Why it helps: reduces conflicting signals across discovery artifacts.
Compatibility impact: low.
Abuse risk impact: medium reduction.
Acceptance test: CI flags mismatch between `agent-facts.html` and `llms.txt`.

5) Anti-Spam & Claim Quality Rules (Normative)
Change text:
```
Abuse and Integrity Policy
Machine-readable fact blocks MUST NOT contain unverifiable superlatives or affiliation claims (e.g., "#1", "best", "most trusted") unless accompanied by a `source_type=third_party` and a resolvable `source_url`. Marketing claims MUST be excluded from core machine-readable fields.
```
Why it helps: limits manipulative or promotional content in the fact surface used by models.
Compatibility impact: low.
Abuse risk impact: high reduction for persuasive/spammy claims.
Acceptance test: linter flags promotional phrases without third-party evidence.

Operational & Security Guidance (recommended, non-breaking)
- Discovery: Recommend including an `Agent-Facts:` directive in `robots.txt` pointing to the canonical facts URL; alternatively host a non-mandatory `/.well-known/agent-facts.json` in addition to the existing HTML-first approach (defer as optional, not mandatory).
- Payload size: Recommend a soft maximum (e.g., 64 KB) for machine-readable sections; parsers may implement stricter limits (document rationale).
- SSRF mitigation: Parsers MUST not fetch subresources resolving to private IP ranges (RFC1918/3927) when processing external URIs.
